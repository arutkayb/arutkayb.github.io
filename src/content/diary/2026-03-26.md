## Reflections

### Wins
- Built the entire dear-diary pipeline in a single day — from nothing to a working extract → reflect → create workflow, with 38 passing tests and a shipped README. This one felt the most meaningful: it's a tool that helps me realize what I've done and feel proud of it
- Caught and fixed the StartApp ANR through Crashlytics monitoring, traced it to a ContentProvider running before the feature flag, and shipped v4.2.5 with a clean manifest-level fix
- Shop funnel analytics immediately surfaced a key insight: 88.5% of users drop off before products even load — the problem isn't in the purchase flow
- Signal Scout pipeline redesign (12→10 stages) shipped with better signal quality after adding per-repo caps

### Learnings
- The manual reflection process isn't just documentation — it forces awareness of what was actually accomplished, what was missed, and whether the day's direction held. The value is in the process of reflecting, not the artifact it produces
- Android SDKs that register ContentProviders auto-initialize before your Application or Activity code runs — feature flags can't gate them unless you block the provider at the manifest merge level
- GA4 only supports String and long for custom event parameters — booleans need to be sent as 0/1 longs

### Challenges
- In-app purchases have been broken since around v4.2.0 — predates the StartApp fix, still under investigation. A concerning regression that needs focused attention
- Permission friction during diary-review skill development required multiple iterations — inline Python and shell constructs triggered security heuristics, solved by switching to standalone scripts
- Signal Scout results still skew toward Streamlit-dominated clusters despite 5K+ star filtering and per-repo caps

### Patterns
- "Euphorically exhausting" — touching 6 projects in one day is both the blessing and curse of working with agents. High throughput, but the cumulative cognitive load is real
- Strong preference for manual reflection over auto mode — the Q&A process surfaces things that pure LLM synthesis misses, especially around intent and emotional significance
- Giving the diary output a week-long evaluation window before tweaking tone and language — resisting the urge to optimize prematurely

---

## Key Takeaways

1. Building tools that make you reflect on your own work is a different category of productivity — it's not about speed, it's about awareness and pride in what you've done.
2. When an Android SDK auto-initializes via ContentProvider, your feature flags are bypassed entirely — the fix lives in the manifest, not in your code.
3. Funnel analytics can reveal that the real problem is nowhere near where you assumed — 88.5% drop-off at product loading, not at checkout.
4. Agent-powered parallelism across many projects delivers real output, but the "euphorically exhausting" pattern suggests building in deliberate focus blocks rather than maximizing breadth every day.
5. Don't tune the tone of a new tool after one use — give it a week of real data before deciding what needs to change.
